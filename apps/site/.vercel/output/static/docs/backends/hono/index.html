Hono Backend

Hono 4 API backend running on Bun with Drizzle ORM, JWT auth, and Zod validation.

Hono Backend

The Hono backend (apps/api-hono/) runs on port 3333 using Bun as the runtime. It provides a lightweight, type-safe REST API with Hono 4, Drizzle ORM (SQLite), JWT authentication via jose, and Zod validation.

Setup

cd apps/api-hono
bun install
bun run db:migrate
bun run db:seed
bun run dev

The API is now available at http://localhost:3333/api/v1/.

5-File Feature Convention

Every feature in src/features/{name}/ follows a strict 5-file convention:

FileResponsibility
{name}.routes.tsHono route definitions, middleware chain, zValidator calls
{name}.handlers.tsParse request, call service, return apiSuccess / apiError
{name}.service.tsBusiness logic; throws AppError on domain errors
{name}.repository.tsDrizzle ORM queries only — no business logic
{name}.schemas.tsZod validation schemas + inferred TypeScript types

Layer Rules

  • Handlers never call repositories directly — always go through the service
  • Services never access c (Hono Context) — they receive and return plain data
  • Repositories never throw AppError — they return null/undefined, services interpret
  • Schemas are pure Zod — no runtime imports, no side effects

JWT Authentication

Hono uses JWT HS256 via the jose library. The same secret signs and verifies tokens.

TokenTTLStorage
Access token1 hourClient-side (cookie via BFF)
Refresh token7 daysDatabase (sessions table)
2FA temp token10 minutesIn-memory
Password reset1 hourDatabase (password_reset_tokens)

The JWT_SECRET is validated at startup — the server crashes if missing or under 32 characters in production.

Response Format (camelCase)

Hono uses camelCase for auth response fields (unlike the PHP backends which use snake_case):

{
  "success": true,
  "data": {
    "user": { "id": "...", "email": "...", "name": "..." },
    "accessToken": "...",
    "refreshToken": "...",
    "expiresIn": 3600,
    "tokenType": "Bearer"
  },
  "status": 200,
  "request_id": "abc123"
}

Drizzle ORM + SQLite

All table definitions live in src/shared/db/schema.ts. The database is SQLite (data.db at project root).

bun run db:generate    # Generate Drizzle migration from schema changes
bun run db:migrate     # Apply pending migrations
bun run db:push        # Push schema without migration (dev only)
bun run db:seed        # Seed data
bun run db:studio      # Drizzle Studio UI

ID convention: text('id').primaryKey().$defaultFn(() => crypto.randomUUID())

Timestamps are stored as ISO 8601 text strings (SQLite has no native date type).

Zod Validation

Every feature has a {name}.schemas.ts file with Zod schemas. Validation is applied at the route level using @hono/zod-validator:

auth.post(
  '/register',
  zValidator('json', registerSchema, (result, c) => {
    if (!result.success) {
      return apiError(c, 'VALIDATION_ERROR', 'Validation failed', 400,
        result.error.flatten().fieldErrors);
    }
  }),
  async (c) => { /* handler */ }
);

Global Middleware (applied to all routes)

Applied in order in src/index.ts: logger(), prettyJSON(), secureHeaders(), requestContextMiddleware, i18nMiddleware, cors().
