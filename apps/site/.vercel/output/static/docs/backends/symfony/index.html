Symfony Backend

Symfony 8 API backend with BetterAuth bundle, Paseto V4 tokens, and API Platform hybrid architecture.

Symfony Backend

The Symfony backend (apps/api-sf/) runs on port 8001 and provides a REST API using Symfony 8, BetterAuth with Paseto V4 tokens, Doctrine ORM, and API Platform 4 for CRUD resources.

Setup

cd apps/api-sf
composer install
php bin/console doctrine:database:create
php bin/console doctrine:migrations:migrate
symfony server:start --port=8001 --no-tls

The API is now available at http://localhost:8001/api/v1/.

Hybrid Architecture

Symfony uses a hybrid approach where standard CRUD operations go through API Platform, while business logic uses invokable controllers:

Use API PlatformUse invokable controller
CRUD (list, show, create, update, delete)Business logic (subscribe, invite, assign)
Standard filtering + paginationMulti-step operations with side-effects
Resource serialization with GroupsAggregations, analytics, dashboards

API Platform provides 34 operations across 13 resources (Product, Category, Review, Order, Plan, Organization, Subscription, Invoice, TeamMember, UsageRecord, Conversation, ChatMessage, CannedResponse).

Paseto V4 Tokens

Unlike JWT, Symfony uses Paseto V4 tokens via the BetterAuth bundle. Paseto provides:

  • No algorithm confusion attacks — the algorithm is part of the protocol, not the token header
  • Mandatory encryption for local tokens, mandatory signing for public tokens
  • Built-in expiration handling without custom claims
  • Smaller attack surface than JWT (no none algorithm, no RSA/HMAC confusion)

Token lifetimes follow the same pattern as other backends: 1-hour access tokens, 7-day refresh tokens. Password hashing uses Argon2id (Symfony’s default hasher).

Invokable Controllers

Each endpoint is a single invokable controller class with #[Route]:

#[Route('/api/v1/cart', name: 'api_v1_cart_show', methods: ['GET'])]
class ShowCartController extends AbstractController
{
    public function __construct(
        private readonly ApiResponseService $apiResponse,
    ) {}

    public function __invoke(): JsonResponse
    {
        // ...
        return $this->apiResponse->success($data);
    }
}

Security

Firewalls

Defined in config/packages/security.yaml:

  • Public routes: auth endpoints (register, login, refresh, forgot-password)
  • Protected routes: require IS_AUTHENTICATED_FULLY
  • Admin routes: require ROLE_ADMIN via Voters

Voters for RBAC

Symfony uses Voters for authorization instead of middleware:

  • ROLE_ADMIN for admin endpoints
  • Custom Voters for organization-level permissions (viewer, member, admin, owner)
  • Object-level authorization checks in individual controllers

Rate Limiting

The RateLimiterService (src/Feature/Auth/Service/RateLimiterService.php) provides cache-backed rate limiting for auth endpoints. Configuration follows the project-wide rate limiting policy (5 login attempts per 15 minutes, etc.).

CORS

Handled by NelmioBundle (config/packages/nelmio_cors.yaml). Uses FRONTEND_URL environment variable. Never add manual Access-Control-* headers in controllers.

Response Format

All responses (both invokable controllers and API Platform) use the same envelope, wrapped by EnvelopeSubscriber for API Platform responses:

{
  "success": true,
  "data": { "..." },
  "status": 200,
  "request_id": "abc123"
}

Paginated responses include: "meta": { "page": 1, "per_page": 20, "total": 100, "last_page": 5 }
