## Backend Selection

The active backend is determined by the `AUTH_BACKEND` environment variable in the Next.js BFF. You can also switch backends at runtime using a cookie:

```typescript
// The BFF proxy reads the active backend from:
// 1. The 'auth_backend' cookie (runtime switching)
// 2. The AUTH_BACKEND environment variable (default)
```

This means you can run all three backends simultaneously and switch between them per-user or per-request.

## HMAC Configuration

HMAC signing is used between the BFF and Laravel to ensure request authenticity. Both sides must share the same secret.

<Aside type="caution">
  The `BFF_SECRET` must be at least 32 characters long and identical on both the Next.js BFF and the Laravel backend. Symfony and Hono do not currently require HMAC.
</Aside>

The signature includes:
- Timestamp (prevents replay attacks)
- HTTP method
- Request path
- SHA-256 hash of the request body (with sorted keys for determinism)

```
signature = HMAC-SHA256(secret, "timestamp:METHOD:/path:bodyHash")
```

Headers sent on every BFF-to-Laravel request:

| Header | Description |
| ------ | ----------- |
| `X-BFF-Id` | Identifier of the BFF instance |
| `X-BFF-Timestamp` | Unix timestamp of the request |
| `X-BFF-Signature` | HMAC-SHA256 signature |

## Cookie Configuration

The BFF stores auth tokens in HttpOnly cookies. Cookie names are derived from the active backend to allow multi-backend sessions:

| Cookie | HttpOnly | Purpose |
| ------ | -------- | ------- |
| `auth_token_{backend}` | Yes | Access token |
| `refresh_token_{backend}` | Yes | Refresh token |
| `token_expires_at_{backend}` | No | Expiration timestamp (readable by client JS for proactive refresh) |
| `auth_backend` | Yes | Active backend identifier |

The `token_expires_at` cookie is intentionally **not** HttpOnly so that client-side JavaScript can check whether a proactive refresh is needed before making API calls.

## CSS Architecture

Since the migration to **Tailwind CSS v4**, configuration is CSS-first — there is no `tailwind.config.ts`.

Each frontend app uses a three-file CSS architecture:

### Next.js (`apps/web`)

| File | Role |
|------|------|
| `src/app/globals.css` | Entry point — `@import`, `@theme inline` mappings, `@custom-variant dark` |
| `src/styles/theme.css` | Design tokens in oklch (`:root` and `.dark` variables) |
| `src/styles/utilities.css` | Custom utility classes |

### TanStack Start (`apps/web-tanstack`)

| File | Role |
|------|------|
| `src/styles.css` | Entry point — `@import` and Tailwind directives |
| `src/styles/theme.css` | Design tokens with neon accent variables |
| `src/styles/utilities.css` | Custom utilities and keyframes |

:::caution
Never add design tokens directly in `globals.css` or `styles.css` — always put them in `theme.css`.
:::

### Dark Mode

Dark mode uses `next-themes` with the `ThemeProvider` in the root layout. The toggle component is at `src/components/theme-toggle.tsx`.

Dark mode tokens are defined in `theme.css` under the `.dark` selector. Tailwind's `@custom-variant dark (&:is(.dark *))` handles class-based dark mode switching.
