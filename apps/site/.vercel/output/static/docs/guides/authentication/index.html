Authentication

Complete guide to the BFF authentication flow with Server Actions, cookie management, and token refresh

Authentication

Headless Kit uses a BFF (Backend-For-Frontend) pattern where the frontend never contacts backends directly. All auth requests flow through the BFF proxy, which handles token storage in HttpOnly cookies.

Architecture Overview

Browser -> Zustand store -> Server Action -> BFF Client -> /api/v1/* proxy -> Backend API

The BFF proxy at /api/v1/[...path]/route.ts intercepts every request, adds the auth token from cookies, and forwards it to the configured backend (Laravel, Symfony, or Hono).

The BFF Client

All Server Actions that communicate with the backend must use bff-client.ts. This is the single HTTP client for server-to-server requests.

import { bffGet, bffPost, bffPatch, bffDelete } from '../_shared/bff-client';

Available methods:

FunctionUsage
bffGet<T>(endpoint, options?)GET request
bffPost<T>(endpoint, body?, options?)POST request
bffPut<T>(endpoint, body?, options?)PUT request
bffPatch<T>(endpoint, body?, options?)PATCH request
bffDelete<T>(endpoint, options?)DELETE request

For public routes (login, register), pass skipAuth: true:

const response = await bffPost<AuthResponse>(
  '/api/v1/auth/login',
  credentials,
  { skipAuth: true }
);

Critical Rule: Cookie Handling in Server Actions

In a 'use server' context, credentials: 'include' is ignored because the code runs in Node.js, not in a browser. Cookies must be forwarded manually.

// WRONG - credentials: 'include' does nothing server-side
const response = await fetch(url, {
  credentials: 'include', // IGNORED in Node.js
});

// CORRECT - bff-client reads and forwards cookies automatically
const cookieStore = await cookies();
const authToken = cookieStore.get(cookieNames.accessToken);
const headers: Record<string, string> = {};
if (authToken?.value) {
  headers['Cookie'] = `${cookieNames.accessToken}=${authToken.value}`;
}

The bff-client.ts handles this automatically. Never use raw fetch() for BFF requests.

Login

The login flow goes through the Zustand auth store, which calls the loginAction Server Action:

// In the auth store (client-side)
login: async (credentials) => {
  set({ error: null, isLoading: true });
  try {
    const response = await loginAction(credentials);
    set({ user: response.data.user, isLoading: false });
  } catch (err) {
    const message = err instanceof Error ? err.message : 'Login failed';
    set({ error: message, isLoading: false });
    throw err;
  }
},

The Server Action delegates to the backend adapter:

'use server';

export async function loginAction(
  credentials: LoginCredentials
): Promise<ApiResponse<{ user: User; access_token: string }>> {
  const backend = await getCurrentBackend();
  const adapter = getAuthAdapter(backend);
  const response = await adapter.login({
    email: credentials.email,
    password: credentials.password,
  });
  return {
    data: {
      user: toUser(response.user),
      access_token: response.tokens.access_token,
    },
  };
}

The adapter calls the backend, receives tokens, and stores them in HttpOnly cookies via storeTokens(). The BFF proxy also extracts tokens from backend responses and sets cookies automatically.

Register

Registration follows the same pattern:

export async function registerAction(
  data: RegisterData
): Promise<ApiResponse<{ user: User; access_token: string }>> {
  const backend = await getCurrentBackend();
  const adapter = getAuthAdapter(backend);
  const response = await adapter.register({
    name: data.name,
    email: data.email,
    password: data.password,
    password_confirmation: data.password_confirmation,
  });
  return {
    data: {
      user: toUser(response.user),
      access_token: response.tokens.access_token,
    },
  };
}

Logout

Logout calls the backend to invalidate the token, then clears all auth cookies:

export async function logoutAction(): Promise<void> {
  const backend = await getCurrentBackend();
  try {
    const adapter = getAuthAdapter(backend);
    await adapter.logout();
  } catch (error) {
    // Even if backend call fails, clear local tokens
    const adapter = getAuthAdapter(backend);
    await adapter.clearTokens();
  }
}

Getting the Current User

The getCurrentUserAction reads the auth cookie and fetches the user from the backend:

export async function getCurrentUserAction(): Promise<User | null> {
  const backend = await getCurrentBackend();
  const adapter = getAuthAdapter(backend);
  const user = await adapter.getUser();
  if (!user) return null;
  return toUser(user);
}

If the token is expired (401/403), the action returns null instead of throwing.

Token Refresh

Headless Kit uses two refresh strategies:

Proactive Refresh (Edge Middleware)

The Edge middleware at src/proxy.ts inspects the JWT expiration. If the token expires within 5 minutes, it sets an x-bff-refresh-needed header. The BFF proxy reads this header and refreshes the token before forwarding the request.

Reactive Refresh (401 Interceptor)

When the BFF proxy receives a 401 from the backend:

  1. Attempts token refresh using the stored refresh token
  2. Retries the original request with the new access token
  3. Stores the new tokens in cookies
  4. If refresh fails, clears all auth cookies and returns 401

Cookie Structure

CookieHttpOnlyPurpose
auth_tokenYesAccess token (1h max-age)
refresh_tokenYesRefresh token (30 days max-age)
token_expires_atNoExpiration timestamp for client-side UX

The token_expires_at cookie is readable by client JavaScript so the UI can show session indicators without exposing the actual token.

Error Handling

All Server Actions throw typed errors from the BffActionError class:

import { BffActionError } from '../_shared/errors';

try {
  return await bffGet<Data>('/api/v1/protected');
} catch (error) {
  if (error instanceof BffActionError && error.statusCode === 401) {
    redirect('/auth/login');
  }
  throw error;
}

Common error codes:

StatusCodeMeaning
401UNAUTHORIZEDNo valid auth token
401SESSION_EXPIREDRefresh failed
403CSRF_ORIGIN_MISMATCHCSRF check failed
422-Validation error
500-Server error

Multi-Backend Support

The auth backend is determined by the AUTH_BACKEND environment variable (or the auth_backend cookie for per-session switching). Each backend has its own adapter:

BackendPortAdapterToken Type
Laravel8002LaravelAdapterBearer
Symfony8001SymfonyAdapterPaseto V4
Hono3333NodeAdapterJWT HS256

All adapters implement the same AuthAdapter type, so the rest of the codebase is backend-agnostic.
