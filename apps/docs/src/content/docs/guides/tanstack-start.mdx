TanStack Start is an alternative frontend option in Headless Kit. It uses **Vite** for bundling and **TanStack Router** for file-based routing, while implementing the same BFF proxy pattern as the Next.js frontend.

<Aside type="note">
  TanStack Start is a viable alternative to Next.js. Both frontends work with all three backends.
</Aside>

## Overview

| Feature | Next.js | TanStack Start |
| ------- | ------- | -------------- |
| Port | 3001 | 3003 |
| Bundler | Webpack / Turbopack | Vite |
| Router | App Router (file-based) | TanStack Router (file-based) |
| SSR | React Server Components | `Route.useRouteContext()` |
| Auth state | Server actions + BFF client | Zustand store |
| BFF proxy | Built-in proxy route | Built-in catch-all route |
| Backend support | Laravel, Symfony, Hono | Hono (default) |

## How Routing Works

TanStack Start uses **file-based routing** powered by TanStack Router. Routes live in `app/routes/`:

```
app/routes/
├── __root.tsx           # Root layout
├── index.tsx            # Home page (/)
├── auth/
│   ├── login.tsx        # /auth/login
│   └── register.tsx     # /auth/register
├── dashboard.tsx        # /dashboard (protected)
└── api/
    └── v1/
        └── $.ts         # BFF proxy catch-all (/api/v1/*)
```

Routes are type-safe -- TanStack Router generates types for route params, search params, and loaders.

## BFF Proxy

The BFF proxy is implemented as a catch-all route at `app/routes/api/v1/$.ts`. It intercepts all requests to `/api/v1/*` and forwards them to the configured backend:

```typescript
// app/routes/api/v1/$.ts
import { createAPIFileRoute } from '@tanstack/react-start/api';

export const APIRoute = createAPIFileRoute('/api/v1/$')({
  GET: async ({ request }) => {
    const url = new URL(request.url);
    const path = url.pathname.replace('/api/v1', '');
    const backendUrl = process.env.VITE_API_URL;

    const response = await fetch(`${backendUrl}/api/v1${path}`, {
      headers: {
        ...Object.fromEntries(request.headers),
        // Forward auth cookies, add HMAC, etc.
      },
    });

    return response;
  },
  POST: async ({ request }) => {
    // Same pattern for POST requests
  },
});
```

This works the same way as the Next.js proxy (`/api/v1/[...path]/route.ts`) -- the browser calls the BFF, and the BFF forwards to the backend. Tokens are stored in HttpOnly cookies and never exposed to client JavaScript.

## Auth State with Zustand

TanStack Start uses a **Zustand store** for client-side auth state management, as opposed to Next.js which uses server actions and React Server Components:

```typescript
// app/stores/auth-store.ts
import { create } from 'zustand';
// User type is defined inline in each app

type AuthState = {
  user: User | null;
  isAuthenticated: boolean;
  login: (email: string, password: string) => Promise<void>;
  logout: () => Promise<void>;
  fetchUser: () => Promise<void>;
};

export const useAuthStore = create<AuthState>((set) => ({
  user: null,
  isAuthenticated: false,

  login: async (email, password) => {
    const res = await fetch('/api/v1/auth/login', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ email, password }),
    });
    const data = await res.json();
    set({ user: data.user, isAuthenticated: true });
  },

  logout: async () => {
    await fetch('/api/v1/auth/logout', { method: 'POST' });
    set({ user: null, isAuthenticated: false });
  },

  fetchUser: async () => {
    const res = await fetch('/api/v1/auth/me');
    if (res.ok) {
      const data = await res.json();
      set({ user: data.user, isAuthenticated: true });
    }
  },
}));
```

## SSR with Route Context

TanStack Start supports SSR by loading auth data in the route context. This avoids the "flash of unauthenticated content" that can occur with client-only state:

```typescript
// app/routes/dashboard.tsx
import { createFileRoute, redirect } from '@tanstack/react-router';

export const Route = createFileRoute('/dashboard')({
  beforeLoad: async ({ context }) => {
    if (!context.auth.isAuthenticated) {
      throw redirect({ to: '/auth/login' });
    }
  },
  component: Dashboard,
});

function Dashboard() {
  const { auth } = Route.useRouteContext();
  return <h1>Welcome, {auth.user?.name}</h1>;
}
```

The auth context is hydrated on the server and passed to the client, so the initial render already has the correct auth state.

## Differences from Next.js

### Server Components vs Zustand

In **Next.js**, server components and server actions handle most data fetching and mutations. The BFF client helpers (`bffGet`, `bffPost`, etc. from `bff-client.ts`) are used in server actions to forward cookies.

In **TanStack Start**, there are no React Server Components. Instead:
- **Zustand** manages auth mutations (login, logout, register)
- **Route context** provides SSR-hydrated auth state
- **Client-side `fetch()`** calls go through the BFF proxy (`/api/v1/*`)

### Backend Support

- **Next.js** supports all three backends (Laravel, Symfony, Hono) with runtime switching via the `AUTH_BACKEND` environment variable
- **TanStack Start** currently targets the **Hono** backend by default via `VITE_API_URL`

## Getting Started

<Steps>

1. **Scaffold with TanStack Start**

   ```bash
   bunx create-headless-app my-app
   ```

   Select **TanStack Start** when prompted for the frontend option.

2. **Configure environment**

   ```bash
   cp apps/web-tanstack/.env.example apps/web-tanstack/.env
   ```

   Edit `apps/web-tanstack/.env`:

   ```bash
   VITE_API_URL=http://localhost:8003
   VITE_APP_URL=http://localhost:3003
   ```

3. **Start development**

   ```bash
   bun run dev
   ```

   TanStack Start runs on [http://localhost:3003](http://localhost:3003).

</Steps>

## Current Limitations

- **Single backend target**: Unlike Next.js, TanStack Start does not yet support runtime backend switching. It targets one backend via `VITE_API_URL`.
- **Experimental status**: TanStack Start itself is still evolving. Check the [TanStack Start docs](https://tanstack.com/start) for the latest status.
