## Role Model

Headless Kit uses a three-tier role system:

| Role | Slug | Description |
| ---- | ---- | ----------- |
| Admin | `admin` | Full access to all resources and admin panel |
| Moderator | `moderator` | Can manage posts and moderate content |
| User | `user` | Standard authenticated user with read access |

Roles are assigned to users via a many-to-many relationship. A user can have multiple roles.

## Permission Model

Permissions follow a `resource:action` pattern:

```
resource:action
```

Available actions:

| Action | Description |
| ------ | ----------- |
| `create` | Create new resources |
| `read` | View resources |
| `update` | Modify existing resources |
| `delete` | Remove resources |
| `manage` | Grants all actions on the resource |

Permissions are assigned to roles, and users inherit permissions through their roles.

### Default Permissions

| Resource | admin | moderator | user |
| -------- | ----- | --------- | ---- |
| `users` | `manage` | -- | -- |
| `posts` | `manage` | `create`, `read`, `update` | `read` |
| `roles` | `manage` | -- | -- |
| `permissions` | `manage` | -- | -- |

## Backend Implementation

<Tabs>
  <TabItem label="Laravel">
    ### Middleware

    Laravel uses custom middleware for route-level authorization:

    ```php
    // Check role
    Route::middleware('role:admin')->group(function () {
        Route::get('/admin/users', [UserController::class, 'index']);
    });

    // Check permission
    Route::middleware('permission:users:manage')->group(function () {
        Route::post('/admin/users/{id}/roles', [UserController::class, 'assignRole']);
    });
    ```

    ### CheckRole Middleware

    ```php
    // app/Http/Middleware/CheckRole.php
    public function handle(Request $request, Closure $next, string ...$roles): Response
    {
        $user = $request->user();

        foreach ($roles as $role) {
            if ($user->roles()->where('slug', $role)->exists()) {
                return $next($request);
            }
        }

        abort(403, 'Insufficient role');
    }
    ```

    ### CheckPermission Middleware

    ```php
    // app/Http/Middleware/CheckPermission.php
    public function handle(Request $request, Closure $next, string ...$permissions): Response
    {
        $user = $request->user();
        $userPermissions = $user->getAllPermissions();

        foreach ($permissions as $permission) {
            [$resource, $action] = explode(':', $permission);

            $hasPermission = $userPermissions->contains(function ($p) use ($resource, $action) {
                return $p->resource === $resource
                    && ($p->action === $action || $p->action === 'manage');
            });

            if ($hasPermission) {
                return $next($request);
            }
        }

        abort(403, 'Insufficient permissions');
    }
    ```

  <TabItem label="Symfony">
    ### Voters

    Symfony uses voters for authorization decisions:

    ```php
    // src/Security/Voter/PermissionVoter.php
    protected function voteOnAttribute(string $attribute, mixed $subject, TokenInterface $token): bool
    {
        $user = $token->getUser();
        [$resource, $action] = explode(':', $subject);

        foreach ($user->getPermissions() as $permission) {
            if ($permission->getResource() === $resource
                && ($permission->getAction() === $action || $permission->getAction() === 'manage')) {
                return true;
            }
        }

        return false;
    }
    ```

    Usage in controllers:

    ```php
    #[Route('/api/v1/admin/users', methods: ['GET'])]
    public function listUsers(): JsonResponse
    {
        $this->denyAccessUnlessGranted('PERMISSION_CHECK', 'users:read');
        // ...
    }
    ```

## Admin API Endpoints

Both Laravel and Symfony expose admin endpoints for managing users, roles, and permissions:

| Method | Endpoint | Required Permission |
| ------ | -------- | ------------------- |
| `GET` | `/api/v1/admin/users` | `users:read` |
| `GET` | `/api/v1/admin/users/{id}` | `users:read` |
| `POST` | `/api/v1/admin/users/{id}/roles` | `roles:manage` |
| `DELETE` | `/api/v1/admin/users/{id}/roles/{role}` | `roles:manage` |
| `GET` | `/api/v1/admin/roles` | `roles:read` |
| `GET` | `/api/v1/admin/permissions` | `permissions:read` |

## Frontend Permission Checking

Permission helper functions are defined inline in each app for checking permissions on the client:

```typescript
// Permission helpers and User type are defined inline in each app

function AdminPanel({ user }: { user: User }) {
  if (!isAdmin(user)) {
    return <p>Access denied</p>;
  }

  return (
    <div>
      {hasPermission(user, 'users', 'manage') && (
        <UserManagement />
      )}
      {hasPermission(user, 'posts', 'delete') && (
        <PostModeration />
      )}
    </div>
  );
}
```

### hasPermission

```typescript
function hasPermission(user: User, resource: string, action: PermissionAction): boolean
```

Returns `true` if the user has the specified permission. Also returns `true` if the user has `manage` on that resource (since `manage` implies all actions).

### hasRole

```typescript
function hasRole(user: User, roleSlug: RoleSlug): boolean
```

Returns `true` if the user has the specified role.

### isAdmin

```typescript
function isAdmin(user: User): boolean
```

Shorthand for `hasRole(user, 'admin')`.

<Aside type="tip">
  These helper functions are isomorphic -- they work on both server and client. Use them in server actions for authorization and in React components for conditional UI rendering. They are defined inline in each app (not in a shared package).
</Aside>
