## Login Flow

<Steps>

1. **User submits credentials**

   The login form sends email and password to a server action.

2. **Server action calls the BFF proxy**

   ```typescript
   // Server Action (apps/web)
   'use server';
   const result = await bffClient.post('/api/v1/auth/login', {
     email,
     password,
   });
   ```

   The `bffClient` (from `apps/web/src/lib/actions/_shared/bff-client.ts`) automatically forwards cookies and sets the `x-bff-internal` header to bypass CSRF checks for server-to-server calls.

3. **BFF proxy forwards to backend**

   The proxy handler:
   - Adds HMAC signature headers (Laravel)
   - Skips auth header (login is a public route)
   - Forwards to `POST /api/v1/auth/login` on the active backend

4. **Backend validates and responds**

   The backend verifies credentials and returns:

   ```json
   {
     "user": { "id": 1, "name": "John", "email": "john@example.com" },
     "access_token": "eyJ...",
     "refresh_token": "dGhp...",
     "expires_in": 3600,
     "token_type": "Bearer"
   }
   ```

5. **Proxy stores tokens in cookies**

   The proxy extracts tokens from the response and sets three HttpOnly cookies:
   - `auth_token_{backend}` -- the access token (maxAge: 1 hour)
   - `refresh_token_{backend}` -- the refresh token (maxAge: 30 days)
   - `token_expires_at_{backend}` -- ISO timestamp of expiration (not HttpOnly, readable by client)

6. **Client receives the response**

   The browser gets the user object and the cookies are set automatically. The access token is never visible to JavaScript.

</Steps>

## Token Refresh

Headless Kit uses two refresh strategies:

### Proactive Refresh (Middleware)

The middleware checks every incoming request:

```typescript
// Simplified logic from createMiddleware
if (shouldRefreshProactively(token, refreshThreshold)) {
  response.headers.set('x-bff-refresh-needed', 'true');
}
```

When the proxy handler sees this header, it refreshes the token **before** forwarding the request to the backend. The default threshold is 300 seconds (5 minutes before expiration).

### Reactive Refresh (401 Interceptor)

If the backend returns 401, the proxy automatically:

1. Attempts to refresh the token using the refresh token cookie
2. If successful, retries the original request with the new access token
3. Sets updated cookies on the response
4. If refresh fails, clears all auth cookies and returns 401

```
Client ──→ BFF ──→ Backend → 401
                   BFF ──→ Backend /refresh → new tokens
                   BFF ──→ Backend (retry) → 200
           BFF ──→ Client (200 + new cookies)
```

### Client-Side Refresh

The `token_expires_at` cookie is readable by JavaScript, allowing the client to check expiration without making API calls:

```typescript
import {
  isTokenExpiredByTimestamp,
  shouldRefreshByTimestamp,
} from '@/lib/services/token-service'; // app-level token helpers

const expiresAt = getCookie('token_expires_at_laravel');

if (isTokenExpiredByTimestamp(expiresAt)) {
  // redirect to login
}

if (shouldRefreshByTimestamp(expiresAt)) {
  // trigger a refresh before next API call
}
```

## Logout Flow

<Steps>

1. **Client triggers logout**

   A server action calls the BFF proxy:

   ```typescript
   await bffClient.post('/api/v1/auth/logout');
   ```

2. **Proxy forwards to backend**

   The backend invalidates the session/token on its side.

3. **Proxy clears cookies**

   Regardless of the backend response, the proxy deletes:
   - `auth_token_{backend}`
   - `refresh_token_{backend}`
   - `token_expires_at_{backend}`

4. **Client redirects**

   The server action redirects the user to the login page.

</Steps>

## Token Lifecycle

| Token | Storage | Lifetime | Rotation |
| ----- | ------- | -------- | -------- |
| Access token | HttpOnly cookie | 1 hour (default) | On every refresh |
| Refresh token | HttpOnly cookie | 30 days (default) | On every refresh |
| Expiration timestamp | Non-HttpOnly cookie | Same as access token | On every refresh |

<Aside type="caution">
  The access token lifetime is controlled by the backend. The BFF reads the `expires_in` field from the backend response and uses it as the cookie `maxAge`. If the backend doesn't return `expires_in`, the BFF defaults to 3600 seconds (1 hour).
</Aside>

## Multi-Backend Sessions

Because cookie names include the backend suffix (`auth_token_laravel`, `auth_token_symfony`, `auth_token_node`), you can be authenticated against multiple backends simultaneously. The `auth_backend` cookie determines which backend is active for the current session.

Switching backends:
- Navigate to `/laravel`, `/symfony`, or `/hono` -- the middleware sets the `auth_backend` cookie automatically
- Or set `AUTH_BACKEND` in the environment for a static default
