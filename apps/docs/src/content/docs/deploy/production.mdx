## Pre-Deployment Checklist

Use this checklist before deploying Headless Kit to production.

### Secrets

- [ ] Generate strong, unique values for `BFF_SECRET` (32+ characters)
- [ ] Generate strong `BETTER_AUTH_SECRET` for Symfony
- [ ] Generate strong `JWT_SECRET` for Hono
- [ ] Run `php artisan key:generate` for Laravel's `APP_KEY`
- [ ] Configure OAuth client secrets for Google/GitHub
- [ ] Remove all default/example credentials from seed data
- [ ] Store secrets in a secrets manager (not in `.env` files on disk)

### Cookie Security

- [ ] Set `secure: true` on all cookie configurations (requires HTTPS)
- [ ] Verify `sameSite: 'lax'` is set (or `'strict'` if cross-origin requests are not needed)
- [ ] Confirm `httpOnly: true` on access and refresh token cookies
- [ ] Verify `token_expires_at` cookie is the only non-HttpOnly cookie

### CORS Configuration

Configure CORS on each backend to only accept requests from your production domain:

```bash
# Laravel (.env)
FRONTEND_URL=https://app.example.com
SANCTUM_STATEFUL_DOMAINS=app.example.com

# Symfony (.env.local)
FRONTEND_URL=https://app.example.com

# Hono (.env)
FRONTEND_URL=https://app.example.com
```

- [ ] Restrict `FRONTEND_URL` to your production domain(s)
- [ ] Do not use wildcard (`*`) origins in production
- [ ] Verify preflight requests work correctly with HTTPS

### HMAC Signing

- [ ] Verify `BFF_SECRET` is identical on Next.js BFF and Laravel backend
- [ ] Verify `BFF_ID` matches on both sides
- [ ] Consider setting a stricter timestamp window for replay protection (default is typically 5 minutes)

### SSL/TLS

- [ ] Terminate TLS at your reverse proxy (Nginx, Caddy, Cloudflare)
- [ ] Redirect all HTTP traffic to HTTPS
- [ ] Set `Strict-Transport-Security` header
- [ ] Verify cookies are transmitted only over HTTPS (`secure: true`)

### Database

- [ ] Use PostgreSQL or MySQL in production (not SQLite)
- [ ] Run migrations before deploying new code
- [ ] Set up automated backups
- [ ] Use connection pooling if applicable

## Deployment Architecture

### Recommended Setup

```
Internet
  │
  ├── CDN (Cloudflare / Vercel Edge)
  │     └── Static assets, caching
  │
  ├── Reverse Proxy (Nginx / Caddy)
  │     ├── HTTPS termination
  │     ├── Rate limiting
  │     └── Request routing
  │
  ├── Next.js BFF (port 3300)
  │     ├── SSR rendering
  │     ├── API proxy
  │     └── Cookie management
  │
  └── Backend API (port 8002/8001/3333)
        ├── Auth processing
        ├── Business logic
        └── Database access
```

### Environment-Specific URLs

```bash
# Production
NEXT_PUBLIC_APP_URL=https://app.example.com
LARAVEL_API_URL=http://api-internal:8002  # Internal network
```

<Aside type="tip">
  In production, the backend should not be publicly accessible. Only the Next.js BFF should be exposed to the internet. The BFF communicates with the backend over an internal network.
</Aside>

## Security Hardening

### Rate Limiting

Apply rate limiting at the reverse proxy level:

```nginx
# Nginx example
limit_req_zone $binary_remote_addr zone=api:10m rate=30r/m;

server {
    location /api/ {
        limit_req zone=api burst=10 nodelay;
        proxy_pass http://nextjs:3300;
    }
}
```

Apply stricter limits on auth endpoints:

```nginx
limit_req_zone $binary_remote_addr zone=auth:10m rate=5r/m;

location /api/v1/auth/login {
    limit_req zone=auth burst=3 nodelay;
    proxy_pass http://nextjs:3300;
}
```

### Security Headers

Add these headers at the reverse proxy:

```nginx
add_header X-Frame-Options "SAMEORIGIN" always;
add_header X-Content-Type-Options "nosniff" always;
add_header X-XSS-Protection "1; mode=block" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'" always;
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
```

### Token Lifetimes

Review and adjust token lifetimes for production:

```typescript
// Token configuration in your app
// (e.g., apps/web/src/lib/services/token-service.ts)
export const TOKEN_CONFIG = {
  ACCESS_TOKEN_MAX_AGE: 60 * 60,       // 1 hour
  REFRESH_TOKEN_MAX_AGE: 60 * 60 * 24 * 30, // 30 days
  REFRESH_THRESHOLD: 5 * 60,            // 5 minutes
  WARNING_THRESHOLD: 2 * 60,            // 2 minutes
};
```

Consider shorter access token lifetimes (e.g., 15 minutes) for higher-security applications.

## Monitoring

### Health Checks

Each backend should expose a health endpoint for load balancers:

```bash
# Laravel
curl http://localhost:8002/api/health

# Symfony
curl http://localhost:8001/api/health

# Hono
curl http://localhost:3333/api/health
```

### Logging

Configure structured logging for the BFF proxy:

```typescript
// In your BFF proxy route (e.g., apps/web/src/app/api/v1/[...path]/route.ts)
// Configure structured logging for proxy requests:
const logger = {
  info: (msg: string, data?: object) => console.log(JSON.stringify({ level: 'info', msg, ...data })),
  warn: (msg: string, data?: object) => console.warn(JSON.stringify({ level: 'warn', msg, ...data })),
  error: (msg: string, data?: object) => console.error(JSON.stringify({ level: 'error', msg, ...data })),
};
```

Key events to monitor:
- Token refresh failures (may indicate expired sessions)
- CSRF validation failures (may indicate attacks)
- HMAC signature mismatches (may indicate configuration issues)
- Backend timeouts (may indicate performance problems)

### Metrics to Track

| Metric | Why |
| ------ | --- |
| Auth success/failure rate | Detect brute force attacks |
| Token refresh rate | Understand session patterns |
| Proxy latency (p50, p95, p99) | Monitor BFF overhead |
| Backend error rate | Detect backend issues early |
| Active sessions | Capacity planning |

## Performance

### Next.js Build

Use standalone output for smaller Docker images:

```typescript
// apps/web/next.config.ts
export default {
  output: 'standalone',
};
```

### Caching

The BFF proxy sets `Cache-Control: no-store` on all API responses. For static assets, configure caching at the CDN/reverse proxy level.

### Connection Pooling

For PostgreSQL/MySQL in production, use connection pooling:

- Laravel: Configure pool size in `config/database.php`
- Symfony: Use `doctrine.dbal.options.pool` or a PgBouncer/ProxySQL sidecar
- Hono: Configure Drizzle connection pool options
